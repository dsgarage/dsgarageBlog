## 1. Anno 117 と Empire Engine の概要

### Anno シリーズ -- 20年以上の歴史を持つ都市建設ゲーム

Anno シリーズは20年以上の歴史を持つ都市建設・経済シミュレーションゲームだ。Anno 117: Pax Romana では、プレイヤーはローマ帝国の総督として皇帝に送り出され、都市を建設し、生産チェーンを管理し、交易ネットワークを構築する。

> "In 2023, we set out to create the most beautiful Anno of all time."
>
> （2023年、我々は史上最も美しい Anno を作ることを目標に掲げました。）

シリーズ初の試みとして、2つの異なるスタート地域を選択できる。ローマ文化圏に影響を受けた「Faiyum」と、ケルト文化圏に基づく「Albion」だ。プレイヤーは現地の伝統を受け入れるか、ローマの影響力を押し付けるかを選択でき、その決断が都市の外観に直接影響する。

### Empire Engine -- 自社プロプライエタリエンジン

Anno シリーズは Ubisoft Mainz が開発する自社エンジン「Empire Engine」で動作している。

- **Anno 1800**（2019年）以来 PC 向けに提供、MSAA（Multi-Sample Anti-Aliasing：ジャギーを滑らかにするアンチエイリアス手法）をサポート
- **2024年3月**: Anno 1800 コンソール版リリース（エンジン初のコンソール対応）
- **2025年11月**: Anno 117: Pax Romana リリース（PC + コンソール同時発売）

Scherzinger 氏は、Anno 1800 コンソール版から Anno 117 への移行は「タイトなプロセス」だったと率直に語った。プリプロダクションが十分でなく、コンソール対応からすぐに次世代タイトルの制作に入る厳しいスケジュールだったという。

## 2. コンテンツ最適化パイプライン -- 140オブジェクトを36に

### 「Conflict」階層の問題

Anno における建物は「Conflict（コンフリクト）」と呼ばれるデータ構造で管理される。単一のメッシュではなく、複数のサブメッシュやエフェクトを含む階層構造だ。

たとえば標準的な居住用建物であっても、メインの建物メッシュに加えて柱、装飾品、植物、照明器具といったサブコンフリクトが参照の連鎖で繋がっている。アーティストにとってはパーツの再利用がしやすいワークフローだが、レンダリング側からは深い階層と大量の細かいオブジェクトが問題になる。

### Simplygon による自動メッシュマージ

この問題を解決するため、Simplygon を用いたコンテンツ最適化パイプラインを構築した。

1. **収集**: マージ可能なメッシュを特定（アニメーション付きオブジェクトは除外）
2. **マージ + LOD 生成**: 複数メッシュを統合し、最適化された LOD（Level of Detail：カメラ距離に応じて表示品質を切り替える仕組み）を生成
3. **出力**: エンジン用の新しい Conflict ファイルを生成

LOD 生成の結果は段階的になる:
- **LOD 0**: 元のマテリアルとテクスチャをすべて保持
- **中間 LOD**: 段階的にディテールを削減
- **最終 LOD**: **単一マテリアル・単一メッシュ**に最適化（遠景描画用）

> エンドゲームシーンの実測値: **約140オブジェクト → 36オブジェクトに削減**

## 3. 地形適応（Terrain Adaptation）-- 頂点カラーで解決する

### 建物の大型化がもたらす課題

Anno 117 では建物が Anno 1800 より大幅に大型化し、地面の占有率が12%から**20%**に増加した。従来の単純な地形適応（建物の底面を地面にスナップするだけ）では、階段部分に隙間が生じたり、空間が不自然に見えたりする問題が発生した。

### 3つの地形適応方式

Scherzinger 氏は3つのアプローチを紹介した:

1. **全体移動方式**: ベース台座を使い、建物全体を地形に合わせて上下させる
2. **コンバーター方式**: 各頂点の高さを個別に調整する
3. **ピボットポイント方式**: メッシュのピボット位置を地面に配置し、そこを基準に位置を調整

### 頂点カラーによるエンコード

最適化パイプラインでメッシュが統合された後も地形適応を機能させるため、**頂点カラーに適応情報をエンコード**する手法を採用した:

| チャンネル | 用途 |
|:---|:---|
| **Red** | 適応強度（Adaption Intensity） |
| **Green** | ローカルピボットの XZ テクスチャ座標 |
| **Blue** | Z オフセット（0 = 非適応、1 = 適応） |

### リメッシュ時の頂点カラー保持

メッシュ統合時に異なるピボットを持つ頂点が補間されると適応情報が壊れるため、2段階のリメッシュ処理を行う:

1. **同一ピボットのオブジェクトのみをマージ**してリメッシュ
2. リメッシュ済みメッシュを**再パッキング**して最終メッシュに統合

### 頂点シェーダーでの地形適応

```hlsl
// 擬似コード: 頂点シェーダーでの地形適応処理
float intensity = vertexColor.r;  // 適応強度
float2 pivotXZ = decode(vertexColor.g);  // ピボット座標
float zOffset = vertexColor.b;  // Z方向オフセット

// ローカルピボットからハイトフィールドを参照
float terrainHeight = SampleHeightfield(pivotXZ);

// 地形ロックによる全体オフセットを減算（ワールド空間）
float delta = terrainHeight - terrainLockOffset;

// 元のワールド座標と適応後の座標を補間
finalPosition = lerp(originalPosition, adaptedPosition, intensity);
```

## 4. GPU アニメーション -- 50,000 インスタンスを 0.1ms で

### 都市建設ゲーム特有のアニメーション課題

Anno のアニメーションは個々には比較的シンプルだが、その数が膨大だ。エンドゲームでは最大**50,000 のアニメーションインスタンス**が同時に存在する。

### CPU から GPU へのオフロード

従来はCPU上でスケルトン階層を更新し、ボーン変換マトリクスをGPUにコピーしていた。Anno 117 では、スケルトンデータをGPUにコピーし、**コンピュートシェーダーで全マトリクスを並列計算**する方式に移行した。

ただし、他のエンジンと異なり**スキニングのデカップリングは行わない**。大半のアニメーションは同一クリップの繰り返し（群衆の歓声、暖房器具の揺れなど特殊ケースを除く）なので、全フレームのスキニング結果をキャッシュするとメモリが膨大になるためだ。

### メモリ最適化

実際にレンダリングされている（カリングされていない）モデルのみにGPUメモリを割り当てる:

| 指標 | 数値 |
|:---|:---|
| 総アニメーションインスタンス | ~50,000 |
| アクティブ（カリング後） | ~4,500 |
| GPU メモリ使用量 | ~25 MB |
| **GPU 計算コスト（RTX 5070）** | **最大 1.0ms、実使用 ~0.1ms** |

## 5. マテリアルシステム -- バインドレスへの移行

### Anno 1800 からの進化

Anno 1800 では各マテリアルごとにランタイムでコンスタントバッファを生成し、レイアウトがマテリアルごとに異なっていた。Anno 117 では大幅に刷新された。

各マテリアルは**事前定義されたコンポーネントのセット**で構成され、コンポーネントの記述を1回定義した後、各コンポーネントのオフセットをファクターとして管理する。マテリアルロード時にユニークなハンドルを割り当て、インデックステーブルへのオフセットでアクセスする仕組みだ。

### バインドレステクスチャ

Anno 1800 のコンソール版からバインドレステクスチャ（Bindless Textures：テクスチャバインド操作をなくし、インデックスでアクセスする方式）に移行。テクスチャの切り替えコストが実質ゼロになった。

### ドローコール最適化の成果

マテリアル共有とドローコールソートにより:

> エンドゲームシーン: **約36,000ドローコール → 約3,000に削減**

## 6. GPU テクスチャストリーミング

![GPU Texture Feedback スライド](slides/IMG_0901.jpg)
*GPU Texture Feedback のシェーダーコード。`WaveActiveMin` と `WaveActiveCountBits` でウェーブ単位の効率的なフィードバックを実現*

### GPUフィードバック方式

Anno 117 では完全に新しいテクスチャストリーミングシステムを実装した。従来のCPUベースではなく、**GPUテクスチャフィードバック**方式を採用している。

シェーダーがテクスチャアクセス時に `RWByteAddressBuffer` にフィードバックを書き込む。エントリーあたり8バイト（最小MIP用 uint + アクセスカウンター用 uint）で、ウェーブ単位（Wave：GPUの並列実行単位）で集約することでアトミック操作のコストを最小化している。

```hlsl
void TextureFeedbackUniform(in float requestedMipLevel, in int texIndex)
{
    uint flooredMip = (uint)(requestedMipLevel + FEEDBACK_MIP_SHIFT);

    // ウェーブ単位で情報を集約
    uint minWaveMip = WaveActiveMin(flooredMip);
    uint accessCount = (uint)WaveActiveCountBits(true);

    if (WaveIsFirstLane())
    {
        uint offset = NUM_BYTES_PER_TEX_FEEDBACK_ENTRY * texIndex;
        cTextureFeedback.InterlockedMin(offset, minWaveMip);
        cTextureFeedback.InterlockedAdd(offset + 4, accessCount);
    }
}
```

![CPU Part スライド](slides/IMG_0902.jpg)
*CPU側の `StreamedTextureInfo` 構造体。`MostRecentFrame` と `MostRecentLODs[4]` でテクスチャの使用履歴を追跡する*

### CPU側の優先度制御

CPU側では `StreamedTextureInfo` 構造体で各テクスチャの使用履歴を追跡し、ロード優先度を決定する:

- **使用カウンター**とロード済み/リクエスト済みMIPの**差分**で優先度をスコアリング
- 最終リクエスト時刻を記録し、使われなくなったテクスチャは低解像度に降格
- **最低MIPレベルは常にメモリに保持**（ポップイン防止）

### パフォーマンス最適化

![Optimizations スライド](slides/IMG_0905.jpg)
*テクスチャフィードバックの最適化。`[earlydepthstencil]` の強制とZプリパスの活用*

テクスチャフィードバックのバリアコストを削減するため:

1. **`[earlydepthstencil]`を強制**: depth/stencil 書き込みやディスカードがないシェーダーに適用
2. **Zプリパス**: テクスチャフィードバックなしで `[earlydepthstencil]` を適用し、その後にフィードバック付きパスを実行
3. **シャドウパスではフィードバック無効化**: パフォーマンス優先のトレードオフ

## 7. テレインシステム -- ディスプレイスメントとゲームプレイの融合

![Terrain and Environment](slides/IMG_0908.jpg)
*Anno 117 の環境描画。山岳から海岸まで、テレインシステムが支える広大な風景*

### テレイン描画の基本

Anno 117 のテレインはカスケードシャドウマップとノードベースのシャドウシステムを使用。地面マテリアルは最大**3レイヤー**のブレンドに対応する。

![Terrain System スライド](slides/IMG_0910.jpg)
*テレインシステムの概要。テクスチャベースのランタイムサブディビジョンで地形ディテールを表現*

### ディスプレイスメントマッピング

![Displacement Mapping Off](slides/IMG_0912.jpg)
*ディスプレイスメントマッピング OFF の状態。岩肌のディテールが平坦に見える*

テレインには**ディスプレイスメントマッピング**（ハイトマップに基づいてジオメトリを実際に変形させる手法）を適用。テクスチャベースでランタイムにサブディビジョン（ポリゴンの細分割）を行い、カメラに近いほどディテールが増す。

### ゲームプレイとの連動 -- 沼地の開拓

![沼地の環境](slides/IMG_0915.jpg)
*ゲーム内の沼地エリア。プレイヤーの技術開発によって排水され、肥沃な土地に変わる*

特筆すべきは、ディスプレイスメントがゲームプレイと直接連動している点だ。プレイヤーが技術を研究して沼地（Marshes）を排水・開拓すると、ディスプレイスメントで地形が水面下に沈んでいた部分がフェードアウトし、肥沃な土地が現れる。マスク情報を使って草の種類も沼地用から通常の草に変更される。

## 8. プロシージャル草生成 -- 45億頂点を 2ms で

### タイルベースの生成パイプライン

![Grass Generation パイプライン](slides/IMG_0918.jpg)
*草生成の全体パイプライン。Terrain Node → Tile Map (R16_UINT) → Compute Shader → Material ID / Coverage / Flags のビットパッキング → 建物・沼地マスクによるフィルタリング*

草は完全にプロシージャル生成される。テレインをタイル分割し（32x32 or 16x16）、コンピュートシェーダーで処理する:

1. **コンピュートシェーダー**（1スレッド = 1テクセル/タイル）がテレインのブレンドマップをサンプリング
2. 出力は16ビットに**ビットパッキング**: Material ID 1（5bit）+ Material ID 2（5bit）+ Coverage（3bit）+ Flags（3bit）
3. 建物フットプリントと沼地マスクをチェックし、草の生成をフィルタリング
4. **フラスタムカリング**でスクリーン外のタイルをスキップ

![Grass Generation フローチャート](slides/IMG_0920.jpg)
*草生成のフローチャート。LOD ごとに1ディスパッチ → Compute Shader（1スレッド/テクセル/タイル）→ 生成判定 → Indirect Draw Buffer*

### 草のレンダリング

![Grass Rendering パイプライン](slides/IMG_0925.jpg)
*草レンダリングのデータフロー。Data Buffer（Tile ID, Position, Material ID, Coverage）→ Vertex Shader → Procedural Mesh + Wind/Clump Map → Pixel Shader*

レンダリングは**インダイレクトドロー**（GPU駆動の描画コマンド）で実行される:

- **Data Buffer**: Tile ID、タイル内位置（ワールド座標）、Material ID 1/2、Coverage を格納
- **Vertex Shader**: ワールド位置からランダム値を生成 → 回転・草タイプ・プロパティを決定 → プロシージャルメッシュを生成
- **Wind Movement + Clump Map**: スクロールテクスチャで風のアニメーション
- 全LODレベルで**一貫した配置**を維持（LOD切り替え時のポッピング防止）

### パフォーマンス

| 指標 | 数値 |
|:---|:---|
| 頂点数（最高品質） | ~45億 |
| ピクセル処理数 | ~80億 |
| **草レンダリング合計** | **~2ms（RTX系GPU）** |

建物が増えるほど草の生成エリアが減るため、草のコストは建物レンダリングと**逆相関**する。シャドウについてはシャドウマップではなくアンビエントオクルージョンで近接する地面との暗さを表現し、スクリーンスペースシャドウを共有する。

## 9. レイトレーシング GI -- 全てのレイはローマへ通ず

### なぜレイトレーシングが必要か

![Raytracing High 品質の都市](slides/IMG_0930.jpg)
*レイトレーシング High 設定でのローマの都市。間接光が建物の陰影にリアリティを加える*

都市建設ゲームは俯瞰視点のため、建物の入り組んだ路地や軒下にはスカイライトからの青い間接光が届くべきだが、従来のラスタライゼーションベースのGI近似ではこれを正確に再現することが困難だった。レイトレーシングGIにより、直接光だけでなく間接光の反射も物理的に正確にシミュレートできるようになった。

### 3段階の品質設定

![Raytracing Overview テーブル](slides/IMG_0935.jpg)
*レイトレーシング品質設定の一覧表。Low/Medium/High の解像度とデノイジング方式を比較*

| RT品質 | 機能 | RT解像度 | デノイズ解像度 | 備考 |
|:---|:---|:---|:---|:---|
| **Low** | GI | 1/16 (w/4, h/4) | 1/4 (w/2, h/2) | Inline RT, opaque only, max LOD 2, テクスチャサンプリングなし |
| **Medium** | GI | 1/16 (w/4, h/4) | 1/4 (w/2, h/2) | Inline RT, opaque only, max LOD 2 |
| **High** | GI + Specular | フル解像度 | フル解像度 | RT パイプライン使用、全ジオメトリ対応 |

デノイズには **NVIDIA Real-Time Denoiser（NRD）** を使用し、フル解像度でないパスではバイラテラルアップサンプリングを適用する。

### Light Path Guided Culling（LiPaC）

![LiPaC スライド](slides/IMG_0950.jpg)
*LiPaC の2レベルアプローチ。非アクティブオブジェクトのバウンディングボックスを Region Hitbox に統合*

全オブジェクトの BLAS（Bottom-Level Acceleration Structure：レイトレーシングの高速化構造体の下位レベル）をメモリに保持すると約**9.5GB**に達する。これは現実的でないため、**動的なBLAS活性化/非活性化システム**を構築した:

![Activation State スライド](slides/IMG_0945.jpg)
*BLAS活性化ステートマシン。Region hit counter が閾値を超えるとBLAS構築をCPUにリクエスト、Instance hit counter で個別BLASを活性化*

1. 初期状態では非アクティブオブジェクトを**Region Hitbox**（バウンディングボックスの統合）で表現
2. **Region hit counter** が活性化閾値を超えると、BLAS構築コマンドをCPUにリードバック
3. **Instance hit counter** が閾値を超えると、詳細BLASをTLAS（Top-Level AS）に追加
4. ヒット数が**非活性化閾値**を下回ると、TLASから除外しバウンディングボックスに戻す

### Terrain Adaptation と BLAS共有

![Terrain Adaption スライド](slides/IMG_0955.jpg)
*Simplygon パイプラインが異なる地形適応のパーツを単一メッシュにマージする問題と、BLAS共有の戦略*

コンテンツ最適化パイプラインでメッシュを統合した結果、1つのメッシュ内に地形適応するパーツとしないパーツが混在する。ラスタライゼーションでは頂点カラーで処理できるが、レイトレーシングでは各インスタンスごとに異なるBLASが必要になってしまう。

解決策として**サブグループ分割**を採用:
- **非変形パーツ**: 全インスタンスで**BLAS を共有**（メモリ効率大幅向上）
- **地形適応パーツ**: インスタンスごとに個別BLAS構築

### Alpha-Tested Geometry

![Alpha-Tested Geometry スライド](slides/IMG_0960.jpg)
*Low/Medium 設定では Inline RT で any-hit シェーダーなし、アルファテスト済みジオメトリを除外*

Low/Medium 設定では**Inline RT（ソフトウェアベースのレイトレーシング実行）**を使用し、any-hit シェーダーを省略する。アルファテスト付きジオメトリ（植生など）はこれらの設定では除外し、High設定でのみ含める。

### 大幅な簡略化による高速化

Low/Medium設定では以下の簡略化を適用:
- **三角形の交差テストを省略** -- バウンディングボックスのヒットのみで判定
- **シャドウレイ不使用** -- オクルージョン近似で代替
- これにより GI の約**70%の効果**を維持しつつ大幅に高速化

### パフォーマンス数値

BLAS管理（最高品質設定）:

| 指標 | 数値 |
|:---|:---|
| 総 BLAS 数 | ~2,200 |
| 共有 BLAS | ~400 |
| ユニーク BLAS | ~1,000 |
| TLAS インスタンス数 | ~30,000 |

GI コスト:

| 設定 | GI 時間 | デノイズ時間 |
|:---|:---|:---|
| **Low** | ~2ms | — |
| **Medium** | ~2ms+ | — |
| **High** | ~2ms | ~1.6ms |

### コンソール対応

- **PlayStation 5**: Low設定でレイトレーシングGI動作
- **PlayStation 5 Pro**: Medium設定で動作

## 10. 開発プロセスの教訓

セッション終盤で Scherzinger 氏は開発から得た教訓を共有した:

### デバッグ可視化を早期に実装する

> "Implementing the debug visualization early on in the process. We have a lot of good use."
>
> （デバッグ可視化をプロセスの早い段階で実装すること。非常に役に立ちました。）

レイトレーシングのBLAS活性化状態やテクスチャストリーミングの優先度など、目に見えないシステムのデバッグに不可欠だったという。

### 多様なハードウェアでテストする

ウェーブフロント（GPUの実行ユニット）の挙動がGPUベンダーや世代によって大きく異なるため、特定のハードウェアでの最適化が他では逆効果になることがある。できるだけ多くの異なるハードウェアで早期にテストすることが重要だと強調した。

### MSAA との同時対応の難しさ

Anno 117 はレイトレーシングに加えてMSAAもサポートして出荷した。MSAAは1ピクセルあたり最大8サンプルを生成するため、レイトレーシング（通常1サンプル/ピクセル）との組み合わせは設定の組み合わせ爆発を引き起こす。テンポラルデータに依存する多くのシステムとの互換性問題もあり、「大きなチャレンジだった」と率直に語った。

### 努力に見合ったか？

> プレイヤーの約 **1.1%** がレイトレーシング設定を使用

数字だけ見ると少ないが、Scherzinger 氏はチームとして取り組む価値があったとの見解を示した。

## まとめ

本セッションは、都市建設ゲームという特殊なジャンルにおけるグラフィックスエンジニアリングの全体像を、驚くほどの詳細さで公開したものだった。特に注目すべきポイントを整理する:

1. **コンテンツ最適化パイプライン**: Simplygon を活用したメッシュマージと LOD 生成により、エンドゲームシーンのオブジェクト数を 140 → 36 に削減。地形適応情報を頂点カラーにエンコードすることで、最適化後もシームレスな地形フィッティングを維持

2. **GPU 駆動のアーキテクチャ**: アニメーション（コンピュートシェーダー）、テクスチャストリーミング（GPUフィードバック）、草生成（インダイレクトドロー）と、可能な限りGPU側で処理する設計思想が一貫している

3. **レイトレーシングの実用的導入**: 9.5GBに達するBLASを動的に活性化/非活性化するLiPaCシステム、地形適応パーツとの共存のためのBLAS共有戦略など、都市建設ゲーム特有の大量オブジェクト問題に正面から取り組んだ

4. **段階的品質設定**: PS5（Low）→ PS5 Pro（Medium）→ ハイエンドPC（High/Ultra）まで、同一フレームワーク内で品質を段階的にスケールさせる設計

5. **開発プロセスの教訓**: デバッグ可視化の早期実装、多様なハードウェアでのテスト、MSAAとの組み合わせの複雑さなど、実践的なアドバイスが語られた

セッションタイトル「All Rays Lead to Rome」の通り、全てのレイ（光線）がローマの街並みを美しく照らすまでの道のりは、エンジニアリングの地道な積み重ねだった。都市建設ゲームというニッチに見える領域から、リアルタイムレンダリングの最前線が垣間見えるセッションだった。
