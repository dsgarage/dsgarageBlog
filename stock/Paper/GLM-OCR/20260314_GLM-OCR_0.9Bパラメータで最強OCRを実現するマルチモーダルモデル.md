# GLM-OCR: 0.9Bパラメータで最強OCRを実現するコンパクトマルチモーダルモデル

## 概要

本論文は、**わずか0.9Bパラメータ**で文書理解の最高性能を達成するコンパクトマルチモーダルOCRモデル「GLM-OCR」を提案する。CogViT視覚エンコーダ（0.4B）とGLM言語デコーダ（0.5B）を組み合わせ、Multi-Token Prediction（MTP）により1ステップで複数トークンを同時予測することで、スループットを約50%向上させている。OmniDocBench v1.5で**総合1位（94.62）**を達成し、235BパラメータのQwen3-VLや Gemini-3 Proを上回る結果を示した。

### 主な貢献

1. **コンパクトな0.9Bモデルで SOTA**: OmniDocBench v1.5 で総合1位、100倍以上大きい汎用VLMを凌駕
2. **Multi-Token Prediction（MTP）**: 1ステップ10トークン同時予測で約50%のスループット改善
3. **2段階パイプライン**: PP-DocLayout-V3によるレイアウト分析 + 並列領域認識で複雑文書に対応
4. **実用的なデプロイ**: `pip install glmocr` で導入可能、vLLM / SGLang / Ollama 対応

## 1. はじめに（Introduction）

文書理解は、財務報告書・科学論文・契約書・請求書などの視覚的にリッチでレイアウト集約型の文書から知識を抽出・構造化する現代情報システムのコア機能である。

### 従来手法の課題

従来のOCRシステム（Tesseract、EasyOCR、PaddleOCR）はプレーンテキストの転写に焦点を当て、レイアウト解析には手作業のルールに依存していた。単純なシナリオでは有効だが、複雑なレイアウトや多様な文書形式では精度が低下する。

近年のマルチモーダル大規模言語モデル（MLLM）は視覚認識と言語理解を統合するが、モデルサイズが大きく、自己回帰デコードによる計算コスト・推論遅延・メモリ消費が課題となっている。

### GLM-OCR の設計思想

実運用のドキュメントインテリジェンスには、3つの要件を同時に満たす必要がある：

1. **高性能**: 表・数式・コード・印鑑など複雑なコンテンツへの対応
2. **高スループット・低遅延**: 推論の効率性
3. **柔軟な統合とドメイン適応**: 容易なカスタマイズ

GLM-OCR はこれらのシステムレベルの要件に統一的なマルチモーダルフレームワークで対応する。

## 2. アーキテクチャ（Methodology）

### 2.1 モデル全体構成

GLM-OCR は GLM-V エンコーダ・デコーダフレームワークに基づき、3つのコンポーネントで構成される：

| コンポーネント | パラメータ数 | 役割 |
|:---|:---|:---|
| **CogViT 視覚エンコーダ** | 0.4B | 文書画像から高レベルの視覚表現を抽出 |
| **軽量クロスモーダルコネクタ** | - | 視覚特徴を言語埋め込み空間に射影 |
| **GLM 言語デコーダ** | 0.5B | 視覚埋め込みとテキストプロンプトに基づき構造化出力を生成 |

視覚エンコーダが生成した特徴は、言語埋め込み空間に射影されプレフィクストークンとしてデコーダに供給される。デコーダは Markdown や JSON 形式の構造化出力を自己回帰的に生成する。

### 2.2 設計の3つの動機

**動機 1: レイアウト分析の統合**

小規模モデルは複雑なレイアウトの文書処理でハルシネーションや繰り返し生成が発生しやすい。レイアウト分析モジュールを明示的に導入し、複雑なレイアウトを複数の単純なサブ問題に分解することで、性能と安定性を大幅に向上させる。

**動機 2: Key Information Extraction（KIE）の統合**

文書パースと KIE はどちらも視覚入力に条件付けされた構造化生成問題として定式化できる。両タスクを共有生成フレームワークで統合し、タスク固有のプロンプトで出力形式を制御する。

**動機 3: Multi-Token Prediction（MTP）の採用**

標準の LLM デコードは1トークンずつ生成するため、長文の文書生成では非効率。MTP により以下の2つの課題を解決する：

- **推論速度**: k トークンを同時予測し、総デコードステップ数を削減
- **文脈モデリング**: 先読み能力の向上。特にテーブルタグや Markdown 構文など局所依存性の強い構造トークンで、「壊れた」タグが減少

### 2.3 2段階パイプライン

GLM-OCR は2つの主要タスクを統一的な生成定式化のもとでサポートする：

**タスク 1: Document Parsing（文書パース）**

```
[入力画像]
    ↓
PP-DocLayout-V3（レイアウト分析）
    ↓  段落・表・数式の領域検出
[領域ごとの画像クロップ]
    ↓  並列処理
GLM-OCR Core（各領域を認識）
    ↓
Merge & Post Process（読み順復元）
    ↓
[Markdown / JSON 出力]
```

- PP-DocLayout-V3 が文書を意味的に一貫した領域に分解
- 各領域を独立に GLM-OCR Core で処理（並列化可能）
- 後処理で読み順を復元し、Markdown / JSON 形式で出力

**タスク 2: Key Information Extraction（KIE）**

```
[入力画像] + [タスクプロンプト（JSON スキーマ等）]
    ↓
GLM-OCR Core（直接処理）
    ↓
[構造化 JSON 出力]
```

- レイアウトクロップを行わず、画像全体をプロンプトとともに直接入力
- モデルがプロンプトの指示に従い関連する視覚領域に暗黙的にアテンション

### 2.4 Multi-Token Prediction（MTP）の詳細

MTP はメインの予測ヘッドに加え、k 個の共有パラメータ補助ヘッドを付加する。各ヘッドは同じパラメータを共有しつつ、異なる将来のオフセットをモデルする。

| 項目 | 値 |
|:---|:---|
| 学習時の同時予測トークン数 | 10 |
| 推論時の平均生成トークン数 | 5.2 / ステップ |
| スループット改善 | 約50% |
| メモリオーバーヘッド | 共有パラメータにより最小限 |

OCR は本質的に**決定論的タスク**であり、局所依存性が強く明示的な構造的教師信号がある。このため厳密な1トークンずつの自己回帰デコードは非効率であり、MTP が特に有効に機能する。

## 3. 学習パイプライン（Training Recipe）

GLM-OCR の学習は4段階で構成され、視覚と言語の表現アラインメントからタスク固有の最適化まで段階的に進む。

### Stage 1: 視覚エンコーダ学習

大規模な画像テキストペアとグラウンディングデータで視覚エンコーダを学習する。

- 数百億の画像テキストペアで学習
- MIM（Masked Image Modeling）と CLIP の二重目標
- 社内のより大きな ViT からの**知識蒸留**で特徴抽出能力を強化

### Stage 2: 視覚・言語事前学習

| サブステージ | 内容 |
|:---|:---|
| Stage 2.1 | GLM-0.5B を ViT に接続し、画像テキスト・文書パース・VQA データで共同事前学習 |
| Stage 2.2 | MTP 目標を導入し、効率的な構造化生成に適応 |

### Stage 3: 教師あり微調整（SFT）

テキスト認識・数式転写・表構造復元・KIE をカバーするキュレーションされた OCR データセットで微調整。

- MTP は学習時も有効（学習と推論の一貫性を保持）
- データミックスをバランスさせ、特定サブタスクへの過学習を防止

### Stage 4: 強化学習（RL）

GRPO（Group Relative Policy Optimization）を適用し、構造化出力の信頼性とタスク固有の精度を改善する。

**タスク別報酬関数の設計:**

| タスク | 主要報酬 | 追加制約 |
|:---|:---|:---|
| テキスト認識 | 正規化編集距離 | 繰り返しペナルティ |
| 数式認識 | CDM スコア | 構造妥当性チェック |
| 表認識 | TEDS スコア | タグ閉じ検証・構造パース |
| KIE | フィールドF1 | JSON パース検証・欠落/重複ペナルティ |

グローバル正則化として、繰り返し比率ペナルティと不正構造ペナルティも適用される。

## 4. 評価（Evaluation）

### 4.1 Public Benchmarks

GLM-OCR は主要な公開ベンチマークの大半で最高スコアを達成した。

**総合比較（Document Parsing + KIE）:**

| ベンチマーク | GLM-OCR | PaddleOCR-VL-1.5 | Deepseek-OCR2 | MinerU 2.5 | dots.ocr | Gemini-3-Pro | GPT-5.2 |
|:---|:---|:---|:---|:---|:---|:---|:---|
| **OmniDocBench v1.5** | **94.6** | 94.5 | 91.1 | 90.7 | 88.4 | 90.3 | 85.4 |
| OCRBench (Text) | **94.0** | 75.3 | 34.7 | 75.3 | 92.1 | 91.9 | 83.7 |
| UniMERNet（数式） | **96.5** | 96.1 | 85.8 | 96.4 | 90.0 | 96.4 | 90.5 |
| PubTabNet（表） | 85.2 | 84.6 | - | **88.4** | 71.0 | 91.4 | 84.4 |
| TEDS_TEST（表） | **86.0** | 83.3 | - | 85.4 | 62.4 | 81.8 | 67.6 |
| Nanonets-KIE | **93.7** | - | - | - | - | 95.2 | 87.5 |
| Handwritten-KIE | **86.1** | - | - | - | - | 94.5 | 78.2 |

※ Gemini-3-Pro と GPT-5.2 はグレー表記（参考値として除外）

**OmniDocBench v1.5 の詳細分析:**

0.9Bパラメータの GLM-OCR が、最高の Overall スコア（94.62）を達成。直接の競合である PaddleOCR-VL-1.5（94.50）や MinerU2.5（90.67）だけでなく、大規模な汎用VLM である Qwen3-VL-235B（89.15）や Gemini-3 Pro（90.33）も上回った。

サブメトリクスの内訳：

| モデル | パラメータ | Overall | Text Edit↓ | Formula CDM↑ | Table TEDS↑ | Table TEDS-S↑ |
|:---|:---|:---|:---|:---|:---|:---|
| **GLM-OCR** | **0.9B** | **94.62** | 0.040 | 93.90 | **93.96** | **96.39** |
| PaddleOCR-VL-1.5 | 0.9B | 94.50 | **0.035** | **94.21** | 92.76 | 95.79 |
| MinerU2.5 | 1.2B | 90.67 | 0.047 | 88.46 | 88.22 | 92.38 |
| Gemini-3 Pro | - | 90.33 | 0.065 | 89.18 | 88.28 | 90.29 |
| Qwen3-VL | 235B | 89.15 | 0.069 | 88.14 | 86.21 | 90.55 |
| GPT-5.2 | - | 85.50 | 0.123 | 86.11 | 82.66 | 87.35 |

表認識で際立つ性能を示し、Table TEDS（93.96）と Table TEDS-S（96.39）で全モデル中最高スコアを記録。

### 4.2 In-House Benchmarks（実運用シナリオ）

6つの高頻度実世界シナリオで評価を実施。

対応言語: 中国語、英語、フランス語、スペイン語、ロシア語、ドイツ語、日本語、韓国語

## 6. ユースケース（Intended Use Cases）

GLM-OCR は2つの利用パラダイムを提供する。

### 6.1 SDK による文書パース

エンタープライズグレードのワークフロー向け。レイアウト認識・マルチモーダル認識・構造化出力生成を包括的にサポート。段落・表・数式・キーバリューペアが混在する異種文書にも対応。

出力は構造化 Markdown 形式で、論理的な文書階層と構造的関係を保持する。

### 6.2 ベースモデルによる軽量 OCR

プロンプト指示で4つのタスクカテゴリを制御する：

**テキスト認識:**

手書きフォント、多言語コンテンツ、パース歪み、背景ノイズなどの課題があっても、テキスト内容を正確に転写。改行・大文字/小文字・句読点・数値・通貨記号を正しく再構築する。

**表認識:**

階層的な列ヘッダー、セル結合、パーセンテージ値、統計的注釈を含む複雑な表の論理構造を正確に復元。CSV やスプレッドシート形式への直接変換が可能。

**数式認識:**

インラインおよびディスプレイ数式を構文的に正しい LaTeX 式に変換。行列、行列式、多段階の添字などの複雑な2次元レイアウトも高忠実度でパース。

**Key Information Extraction（KIE）:**

プロンプトで JSON スキーマを明示的に指定すると、モデルが指定形式に準拠した出力を生成。密な税関申告書などからネストされた JSON エンティティを正確に抽出。

## 7. 制限事項（Limitations）

### 7.1 2段階アーキテクチャの制約

レイアウト分析 → 領域認識のパイプラインでは、レイアウト検出の誤りが下流の認識性能に伝播する可能性がある。ページ間依存関係や不規則な複数カラム構造では、読み順の復元が不完全になる場合がある。

### 7.2 データカバレッジの限界

以下のシナリオでは性能低下が発生する可能性がある：

- 極度に低解像度・著しく歪んだ文書
- 非常に複雑な数式
- 密集または不規則な表構造
- 学習コーパスで過小に表現された言語

### 7.3 構造化出力のばらつき

生成モデルとして、改行や空白の扱いに微小な確率的変動が生じることがある。強化学習と構造的教師信号により緩和されているが、厳密なフォーマット保証は完全にはできない。

### 7.4 KIE の制約

プロンプトベースの KIE では、抽出精度がプロンプトの仕様とスキーマの明確さに依存する。暗黙的または曖昧なフィールド境界を持つ複雑なフォームでは、不完全または冗長な出力が発生する場合がある。

---

## 8. 結論（Conclusion）

GLM-OCR は、大規模モデルスケーリングに依存せず、制御可能なレイテンシ・メモリ効率・構造化出力の信頼性を優先した設計により、実運用の文書理解を実現するソリューションである。

レイアウト認識前処理とマルチトークンデコードの組み合わせにより、多様な文書タイプで競争力のある認識精度を維持しつつ、スループットと安定性を改善した。モデルアーキテクチャ・デコード戦略・タスク構造の慎重なアラインメントが、パラメータ規模を増大させることなく大幅な効率向上をもたらすことを示している。

今後は、極端なレイアウト複雑性下での頑健性改善、多言語カバレッジの拡充、構造化出力の一貫性強化に焦点を当てるとしている。
