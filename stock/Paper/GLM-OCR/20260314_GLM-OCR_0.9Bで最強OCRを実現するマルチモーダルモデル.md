## なぜ注目か

- **0.9Bパラメータで OmniDocBench v1.5 総合1位（94.62）** — Qwen3-VL-235B（89.15）やGemini-3 Pro（90.33）を上回る
- エッジデプロイ可能な軽量さと、大規模モデルを超える精度の両立
- Multi-Token Prediction（MTP）で約50%のスループット改善
- `pip install glmocr` で即利用可能な実用性

## アーキテクチャ

```
[入力画像]
    ↓
CogViT Visual Encoder（0.4B）
    ↓
軽量クロスモーダルコネクタ
    ↓
GLM Language Decoder（0.5B）+ MTP（10トークン同時予測）
    ↓
[Markdown / JSON 出力]
```

### 2段階パイプライン（Document Parsing）

1. **PP-DocLayout-V3** によるレイアウト分析（段落・表・数式の領域検出）
2. **領域ごとの並列認識** → Merge & Post Process で読み順を復元

### Multi-Token Prediction（MTP）

- 標準の自己回帰デコードは OCR のような決定的タスクでは非効率
- 1ステップで **10トークンを同時予測**（推論時は平均5.2トークン/ステップ）
- 共有パラメータ方式でメモリオーバーヘッドを抑制
- **約50%のスループット向上**

## 学習パイプライン（4段階）

| Stage | フェーズ | 内容 |
|:---|:---|:---|
| Stage 1 | Vision Encoder 学習 | 数百億の画像テキストペアで MIM + CLIP 目標 |
| Stage 2.1 | VL Pretrain | GLM-0.5B を接続し、文書パース・VQA で事前学習 |
| Stage 2.2 | Pretrain + MTP | MTP 目標を導入 |
| Stage 3 | SFT + MTP | テキスト/数式/表認識・KIE で教師あり微調整 |
| Stage 4 | RL（GRPO） | タスク別報酬関数で強化学習 |

### Stage 4 の報酬設計

| タスク | 主要報酬 | 追加制約 |
|:---|:---|:---|
| テキスト認識 | 正規化編集距離 | 繰り返しペナルティ |
| 数式認識 | CDM スコア | 構造妥当性チェック |
| 表認識 | TEDS スコア | タグ閉じ検証・構造パース |
| KIE | フィールドF1 | JSON パース検証・欠落/重複ペナルティ |

## 導入方法

```bash

pip install glmocr

# フル機能（ローカル推論含む）
pip install "glmocr[selfhosted]"

pip install "glmocr[server]"
```

```bash
glmocr parse examples/source/code.png
```

```python

result = parse("image.png")
```

## 制限事項

1. **2段階アーキテクチャの制約**: レイアウト検出の精度に依存
2. **データカバレッジ**: 一部言語・文書タイプでのカバー不足
3. **構造化出力のばらつき**: 複雑なネスト構造で不安定になる場合あり
4. **KIE**: フィールド定義が曖昧な場合に精度低下
