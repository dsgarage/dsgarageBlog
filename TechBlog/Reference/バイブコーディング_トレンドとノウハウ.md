## バイブコーディングとは何か？

### 定義

**バイブコーディング（Vibe Coding）** とは、プログラマが実現したい機能やアプリの「雰囲気（vibe）」や方向性を自然言語でAIに伝え、コードの具体的な実装はAIに任せて進める新しいコーディングスタイル。

### 特徴

| 従来のコーディング | バイブコーディング |
|:---|:---|
| 開発者がコードを書く | AIがコードを書く |
| 仕様書を詳細に作成 | 直感的な指示で進める |
| 実装に時間がかかる | 短期間で形になる |

### 背景

- 2025年初頭に **OpenAIのAndrej Karpathy氏** が提唱
- 2025年3月にはWikipediaにもエントリが作成
- 「ノーコード・コーディング」とも称される新興トレンド

### 人間の役割の変化

> コードそのものを書く作業から、**何を作るか・どう作るかを設計しAIに教える作業**へと、人間の役割がシフトしている

AIは「書く」を担当し、人間は：
- ゴール設定
- 制約条件の定義
- アーキテクチャの判断
- 品質のレビュー

を担当する。

## Claudeによるコーディング支援

### Claude Codeとは

Anthropic社の開発エージェント。

**特徴:**
- プロジェクトディレクトリ内のファイル構成を**すべて読み込み、内容を把握**
- 自然言語で質問・指示すると、**プロジェクト全体の文脈を踏まえて回答**
- **複数ファイルにわたる変更を提案・実施**
- Git操作やビルドコマンドの実行も可能

> まさに **有能なジュニアプログラマ** がフォルダ内に住み着いたような体験

### CLAUDE.mdファイル

プロジェクト内に `CLAUDE.md` を配置し、プロジェクト固有のルールを記載する。

```markdown
## Code Style（コードスタイル）
- TypeScriptを全てに用いる（例外なし）
- Reactは関数コンポーネント + フックのみ使用
- インデントはスペース2つ
- 変数名はcamelCase、コンポーネント名はPascalCase

## Architecture Notes（アーキテクチャ方針）
- 状態管理にはZustandを使用
- API呼び出しは /src/utils/api.ts のラッパー経由
- 新規コンポーネントにはテストを付属させる
- パフォーマンス重視（バンドルサイズに留意）

## Don't Do（禁止事項）
- クラスコンポーネントは使わない
- エラー境界の仕組みをバイパスしないこと
- 1ファイル500行を超えるコンポーネントを書かないこと
```

### プランモード（Plan Mode）

**Shift+Tab を2回押す** ことで切り替え可能。

**特徴:**
- コードを実行・変更せずに、**必要な実装プランを分析・提案**
- 複雑な機能追加や大規模リファクタ前に使用
- 設計プランを出させ、承認・修正してから実装に移る

**使い方:**
1. 新機能の要件をプランモードで説明
2. Claudeが設計プランを提示
3. 人間がレビュー・修正
4. プランモードを解除して実装開始

## プロンプトの書き方

### 基本原則

単に「○○を作って」ではなく、以下を含めて伝える：

1. **何のためにその機能が必要なのか**（目的）
2. **達成したいユーザー体験**（ゴール）
3. **性能や条件の制約**（制約）

### 良いプロンプトの例（Unity FPSゲーム）

```
UnityでRigidbodyベースのプレイヤーコントローラを作って。

含める機能:
- 移動・ダッシュ・ジャンプ・しゃがみ
- 重力・スライディング
- 一人称視点操作

追加で作成:
- InputManagerスクリプト（旧Inputシステム使用）
- GunInterfaceインタフェース（reload/secondary fire/primary fire）
- Gunクラス（フルオート/セミオート切替可能）
- Projectileスクリプト（衝突時にデバッグログ出力して消滅）

注意点:
- モジュール化された正しい設計で
- 手抜きはしないこと
```

### プロンプトのポイント

| ポイント | 説明 |
|:---|:---|
| 具体的に書く | 抽象的な指示は期待外れの結果になる |
| 制約を先に示す | 後から「やり直し」を減らす |
| 目的を伝える | AIが適切な判断をしやすくなる |
| 禁止事項を明示 | 不要な実装を防ぐ |

## 他のAIとの比較

### ChatGPT（GPT系）

| 項目 | ChatGPT | Claude Code |
|:---|:---|:---|
| 提供形態 | 対話ベース | プロジェクト統合型 |
| ファイル操作 | プラグイン必要 | 自動で読み書き |
| コンテキスト長 | 32kトークン | 100kトークン級 |
| 得意分野 | コード解説、スニペット生成 | プロジェクト全体の変更 |

**活用法:** Claude Codeで生成した変更内容についてChatGPTにも見解を求め、**複数AIのクロスレビュー**で品質を高める

### GitHub Copilot

| 項目 | Copilot | Claude Code |
|:---|:---|:---|
| 統合先 | IDE（VS Code等） | CLI |
| 動作方式 | リアルタイム補完 | 対話型エージェント |
| コンテキスト | 編集中ファイル周辺 | プロジェクト全体 |
| 得意分野 | 定型コード、ボイラープレート | 大規模改変、設計 |

**比喩:**
- Copilot = 「文章の続きをその場で執筆してくれる優秀な秘書」
- Claude Code = 「設計図を描き現場監督もしてくれる共同開発者」

**活用法:** 日常的なコーディングはCopilotで効率化し、大きな機能追加やリファクタはClaudeで対話しながら進める

## ベストプラクティス

### AIを「有能な新人」としてマネジメントする

#### 1. 目的（ゴール）を最初に明確に伝える

```
悪い例: 「家具配置機能を作って」

良い例: 「この機能はユーザーが迷わず家具を配置できるようにするためのもの。
        ロード時間は3秒以内が必須条件。」
```

#### 2. 制約条件・禁止事項を先に示す

```
「バックエンドはCloud Runに固定し、外部の有料APIは使わないこと」
```

#### 3. タスクを細かく分割し短いサイクルで検証する

- 理想は **2〜3日（あるいは数時間～半日）のスプリント**
- 中間成果物を必ず確認
- 長期間放置すると「ゴールとは違う山を登り始める」

```
「次のチェックポイントは木曜午前で、その時点で○○が完了していること」
```

#### 4. 常にプランを立ててから実装させる

- いきなりコードを書かせない
- プラン → レビュー → 実行の流れを徹底
- 「まず設計方針を考えて」「手を動かすのは待って」

#### 5. AIに質問させる

```
「実装開始前に不明点は遠慮なく聞いて」
```

AIからの確認質問例：
- 「SNSログインは必要ですか？」
- 「ユーザ設定はDBに保存しますか？」

#### 6. 成果をほめ、修正点は建設的に伝える

```
良い例: 「助かった！ただ目的に沿うにはこちらの方向が良さそう」

悪い例: 「違う。やり直して」
```

### サーファーとライフセーバーの関係

> AI（サーファー）には自由に波に乗ってもらい、溺れそうになったら人間（ライフセーバー）がさっと引き戻す

## 注意点と落とし穴

### AIの典型的な癖

| 癖 | 説明 |
|:---|:---|
| 冗長な実装 | 必要以上にコードを書こうとする |
| ルールの忘却 | 一度与えた方針やルールを無視することがある |
| 表面的な理解 | 設計全体を深く理解しているわけではない |

### 「意外と動くスパゲッティ」問題

- AIの生成物は「動くこと」が多い
- 問題が表面化しづらい
- 気付いた時には技術的負債を抱えている

### 実例：ジュニアエンジニアの失敗

| 項目 | 数値 |
|:---|:---|
| 開発速度 | 2〜3倍に向上 |
| レビュー負担 | 5倍以上に増加 |

**原因:**
- AIが「全てを把握して直してくれている」と錯覚
- つい承認（Enterキー）を連打
- 警戒心が薄れた

### 解決策

1. **タスクを細かく分割**
2. **自分の理解の範囲で作業を進める**
3. **中間成果物を必ずレビュー**
4. **「まず説明して」で内容を確認してから承認**

## 適用領域と事例

### 向いている領域

| 領域 | 理由 |
|:---|:---|
| **プロトタイプ開発** | 短期間で形にできる |
| **ボイラープレート生成** | 定型コードを素早く作成 |
| **リファクタリング** | 大規模な変更を提案 |
| **学習・実験** | 新技術の素早い試行 |

### 慎重さが必要な領域

| 領域 | 理由 |
|:---|:---|
| **本番環境のコア機能** | 品質保証が難しい |
| **セキュリティ関連** | 脆弱性のリスク |
| **複雑なビジネスロジック** | AIの理解が表面的 |

### 成功事例

- **4時間でWebサイト構築**: InfoWorldの開発者がAstroアプリを構築
- **24時間でFPSゲーム開発**: Unity製ゲームをAIと共に完成

### バイブコーディングの本質

| 要素 | 従来 | バイブコーディング |
|:---|:---|:---|
| コードを書く人 | 人間 | AI |

3. **短いサイクルで検証**
4. **AIを「有能な新人」としてマネジメント**
5. **盲信せず、必ずレビュー**
