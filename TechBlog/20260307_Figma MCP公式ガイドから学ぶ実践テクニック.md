# Figma MCP公式ガイドから学ぶ実践テクニック

以前の記事「[Figma MCPでデザインからUnity UIを生成する](20260303_Figma MCPでデザインからUnity UIを生成する.md)」では、Figma MCPの基本的な使い方と、`/simplify` + `/batch` による複数画面の横展開を解説しました。

その後、Figmaが公式に[MCP Server Guide](https://github.com/figma/mcp-server-guide)を公開しました。読み込んでみると、前回の記事では触れなかった重要な情報が大量にあることが分かりました。

**月6回のレート制限**、**13種類のツール**、**Code Connectによるコンポーネント再利用**、**大規模デザインの2段階取得**——知らないと確実に壁に当たるポイントばかりです。

本記事では、公式ガイドの内容と前回記事の差分にフォーカスし、実践で役立つテクニックを整理します。

---

## まず知るべき: レート制限

前回の記事では触れていなかった、最も重要な制約です。

| プラン | 制限 |
|:---|:---|
| **Starter / View / Collab** | **月6回**まで |
| **Dev / Full（Professional以上）** | Tier 1 REST APIと同等の分単位制限 |

Starterプランでは月にたった6回しかMCPツールを呼び出せません。`get_design_context` を1画面で1回、`get_screenshot` で1回と数えると、**3画面で上限に達します**。

### 対策

1. **プランを確認する**: `whoami` で自分のシート種別（View / Collab / Dev / Full）を確認
2. **呼び出しを最小化する**: まず `get_metadata` でノード構造を把握し、必要なノードだけ `get_design_context` を呼ぶ
3. **Dev / Full シートを検討する**: 本格運用する場合は分単位制限のプランに移行する

```
> Figma MCPのwhoamiを実行して

結果:
{
  "email": "your@email.com",
  "plans": [{ "name": "Your Team", "seat": "Full", "tier": "professional" }]
}
→ Full シートなので分単位制限（実質無制限に近い）
```

## セットアップ: Plugin方式が推奨に

前回の記事では `claude mcp add` による手動セットアップを前提としていました。公式ガイドでは**Plugin方式**が推奨されています。

### Claude Code（推奨）

```bash
claude plugin install figma@claude-plugins-official
```

これだけで以下が自動設定されます。

- MCP サーバー設定
- エージェントスキル（デザイン実装、Code Connect、デザインシステムルール生成）
- アセット取扱いルール

### 手動セットアップ（従来方式）

```bash
claude mcp add --transport http figma https://mcp.figma.com/mcp
```

### VS Code

```
⌘ Shift P → MCP:Add Server → HTTP → https://mcp.figma.com/mcp
```

GitHub Copilotが有効なアカウントが必要です。

### Cursor

```
/plugin-add figma
```

Figma Pluginがインストールされ、MCP設定・スキル・ルールが自動で追加されます。

## 全13ツール解説

前回の記事では3ツールしか紹介していませんでした。公式ガイドには**13ツール**が記載されています。

### デザイン読み取り系（5ツール）

| ツール | 用途 | 対応ファイル |
|:---|:---|:---|
| `get_design_context` | デザインの構造・スタイル・コードヒントを取得（React+Tailwind形式） | Design, Make |
| `get_screenshot` | ノードのスクリーンショットを取得 | Design, FigJam |
| `get_variable_defs` | デザイントークン（色・余白・フォント等の変数）を取得 | Design |
| `get_metadata` | ノードのXMLメタデータ（ID・名前・位置・サイズ）を取得 | Design |
| `get_figjam` | FigJamダイアグラムのメタデータ + スクリーンショットを取得 | FigJam |

### Code Connect 系（4ツール）

| ツール | 用途 |
|:---|:---|
| `get_code_connect_map` | Figmaノード ↔ コードコンポーネントの対応マップを取得 |
| `add_code_connect_map` | 新しいマッピングを追加 |
| `get_code_connect_suggestions` | マッピング候補を自動検出・提案 |
| `send_code_connect_mappings` | 提案されたマッピングを確定 |

### 生成系（2ツール）

| ツール | 用途 |
|:---|:---|
| `generate_figma_design` | WebページをキャプチャしてFigmaデザインに変換 |
| `generate_diagram` | Mermaid記法からFigJamダイアグラムを生成 |

### ユーティリティ（2ツール）

| ツール | 用途 |
|:---|:---|
| `create_design_system_rules` | デザインシステムルールファイルを自動生成 |
| `whoami` | 認証情報・プラン・シート種別を確認 |

## 大規模デザインの2段階取得

前回の記事では「URLを渡して `get_design_context` を呼ぶ」という1ステップでした。公式ガイドでは、大きなデザインに対して**2段階取得**が推奨されています。

### なぜ必要か

`get_design_context` は選択範囲のデザインデータを全て返します。画面全体を指定すると:

- レスポンスが巨大になりコンテキストを圧迫する
- 処理が遅くなる、またはエラーになる
- 不完全なレスポンスが返る場合がある

### 2段階取得の手順

```
Step 1: get_metadata でノードマップを取得
├── ページ全体のノード構造をXMLで取得
├── レイヤーID・名前・種類・位置・サイズが分かる
└── コンテキスト消費が少ない

Step 2: get_design_context で必要なノードだけ取得
├── Step 1 で特定したノードIDを指定
├── 必要な部分だけの詳細データを取得
└── レート制限の消費も最小化
```

### 実践例

```
> まず get_metadata でこのページの全ノード構造を取得して。
  https://www.figma.com/design/XXXXX/MyApp?node-id=0-1

→ XMLで全レイヤーの一覧が返る

> 「LoginForm」と「HeaderNav」のノードだけ
  get_design_context で取得して実装して。
```

これにより、不要なノードのデータ取得を避け、レート制限の消費を抑えられます。

## Code Connect: コンポーネント再利用の本命

前回の記事では「ヒントにCode Connectマッピングが含まれる」と触れた程度でした。公式ガイドでは、Code Connectを**コンポーネント再利用の最善手**と位置づけています。

### Code Connect がないとどうなるか

> Code Connect keeps your generated code consistent with your codebase.
> Without it, the model is guessing.

AIはFigmaのデザインからコードを生成しますが、**プロジェクトに既存のコンポーネントがあることを知りません**。結果として:

- 既存の `Button` コンポーネントがあるのに、新しいボタンを生成してしまう
- デザインシステムのトークンを無視したハードコード値が出力される
- プロジェクトのコーディング規約と異なるコードが生成される

### Code Connect のワークフロー

```
Step 1: get_code_connect_suggestions
├── Figmaコンポーネントとコードベースのコンポーネントを
│   自動で照合し、マッピング候補を提案
└── 例: Figmaの「PrimaryButton」→ src/components/Button.tsx

Step 2: ユーザーが提案をレビュー

Step 3: send_code_connect_mappings
└── 確認済みのマッピングを確定

Step 4: 以降の get_design_context で自動的に
        マッピング済みコンポーネントが使われる
```

### 手動でマッピングを追加する場合

```
> add_code_connect_map で、Figmaの「CardComponent」を
  src/components/ui/Card.tsx にマッピングして。
```

### Unity開発での活用

Unity UI Toolkit のコンポーネント（カスタム VisualElement）がある場合、Code Connect でFigmaコンポーネントとマッピングしておけば、`get_design_context` が既存コンポーネントの使用を前提としたコードを返します。

## アセット取扱いルール（CLAUDE.mdに追記すべき）

公式ガイドで強調されているのに、前回の記事で触れていなかったルールです。

```markdown
## Figma MCP アセットルール

- Figma MCPがlocalhost URLで画像/SVGを返した場合、そのソースを直接使用すること
- 新しいアイコンパッケージを追加してはならない（アセットはFigmaのペイロードに含まれる）
- localhost ソースが提供されている場合、プレースホルダーを作成してはならない
```

Figma MCPはアセット配信用のローカルエンドポイントを持っています。AIが「画像がないから仮のプレースホルダーを作ろう」とするのを防ぐために、このルールが必要です。

## 公式推奨の実装フロー（6ステップ）

前回の記事では「URLを渡す → 変換 → /simplify」の3ステップでした。公式ガイドでは**6ステップの必須フロー**が推奨されています。

```
1. get_design_context でデザインデータを取得
2. レスポンスが大きすぎる場合は get_metadata → 必要ノードだけ再取得
3. get_screenshot で視覚的リファレンスを取得
4. 必要なアセットをダウンロード
5. プロジェクトの規約・フレームワークに合わせて実装
6. Figmaデザインと1:1で一致するか検証
```

前回の記事との主な違い:

| ステップ | 前回の記事 | 公式推奨 |
|:---|:---|:---|
| メタデータ取得 | なし | 大きなデザインでは必須 |
| スクリーンショット | オプション扱い | 毎回取得を推奨 |
| アセットDL | 言及なし | 実装前に必ずDL |
| 検証 | `/simplify` に委任 | 「1:1の視覚的一致」を明示 |

## Figmaファイルの構造化ベストプラクティス

公式ガイドには、**Figma側の準備**に関する推奨事項があります。AIが正確にデザインを読み取るために、デザイナー側で以下を実践すべきです。

| 項目 | 推奨内容 |
|:---|:---|
| **コンポーネント化** | 再利用する要素（ボタン、カード、入力欄等）は必ずコンポーネントにする |
| **変数の使用** | 色・余白・角丸・タイポグラフィに変数を使う |
| **レイヤー名** | セマンティックに命名する（`CardContainer` ○ / `Group 5` ✕） |
| **Auto Layout** | レスポンシブ意図を伝えるために使う。リサイズして動作確認してからコード生成する |
| **アノテーション** | 振る舞い・配置・レスポンスなど、視覚だけでは伝わらない意図を記載する |

デザイナーがこれらを実践すれば、`get_design_context` の出力品質が大幅に向上します。

## generate_diagram: FigJamにダイアグラムを生成

前回の記事では未紹介だったツールです。Mermaid記法からFigJamにダイアグラムを直接生成できます。

```
> Figma MCPのgenerate_diagramで、ユーザー認証フローの
  フローチャートを作成して

→ FigJamにダイアグラムが生成され、URLが返される
```

対応するダイアグラム種別:
- フローチャート（flowchart / graph）
- シーケンス図（sequenceDiagram）
- 状態遷移図（stateDiagram）
- ガントチャート（gantt）

技術ドキュメントやアーキテクチャ図を素早く作成したい場合に便利です。

## generate_figma_design: WebページをFigmaに取り込む

もう1つの未紹介ツールです。WebページをキャプチャしてFigmaデザインに変換します。

```
> ローカルの開発サーバーのUIをFigmaに取り込んで

> このログインページをFigmaファイルに追加して
  https://www.figma.com/design/XXXXX/MyApp
```

既存のWebアプリのUIをリデザインする際や、現状のUIをFigma上で分析する際に使えます。

---

## create_design_system_rules: ルールファイル自動生成

AIがデザインシステムに沿ったコードを生成するためのルールファイルを自動生成するツールです。

```
> create_design_system_rules を実行して、
  このプロジェクトのデザインシステムルールを生成して
```

生成されたルールを `.claude/rules/` や `.cursorrules` に保存すれば、以降のコード生成でデザインシステムが自動的に適用されます。

## 前回記事からの CLAUDE.md 更新例

前回の記事で示したCLAUDE.mdルールを、公式ガイドの内容で更新します。

```markdown
## Figma MCP 利用ルール

### 必須フロー（スキップ禁止）
1. `get_design_context` でデザインデータを取得する
2. レスポンスが大きい・不完全な場合は `get_metadata` で概要を取得し、
   必要なノードだけ `get_design_context` で再取得する
3. `get_screenshot` で視覚的リファレンスを取得する
4. アセット（画像・SVG）をダウンロードしてから実装を開始する
5. プロジェクトの規約・フレームワークに合わせてコードを生成する
6. Figmaデザインと1:1で一致するか検証する

### アセットルール
- Figma MCPがlocalhost URLで返した画像/SVGはそのまま使用すること
- 新しいアイコンパッケージを追加してはならない
- プレースホルダーの作成は禁止（localhost ソースがある場合）

### URL の読み方
- `figma.com/design/:fileKey/:fileName?node-id=:nodeId`
  - nodeId の "-" は ":" に変換して使う
- `figma.com/design/:fileKey/branch/:branchKey/:fileName`
  - branchKey を fileKey として使う

### 出力形式
- UI Toolkit（UXML + USS）を優先する
- uGUI が必要な場合は明示的に指示する
- ハードコード値を避け、デザイントークンを変数参照する
- 既存コンポーネントを再利用する（Code Connect がある場合）
```

## まとめ

| 項目 | 前回の記事 | 今回の追加内容 |
|:---|:---|:---|
| **レート制限** | 未記載 | Starterは月6回。Dev/Fullは分単位 |
| **セットアップ** | 手動（`mcp add`） | Plugin方式推奨（`claude plugin install`） |
| **ツール数** | 3ツール | 13ツール（Code Connect 4つ、生成系 2つ等） |
| **大規模デザイン** | 1ステップ取得 | `get_metadata` → `get_design_context` の2段階 |
| **Code Connect** | 言及のみ | 4ツールのワークフロー解説 |
| **アセット** | 未記載 | localhost画像をそのまま使うルール |

| **Figma側の準備** | 未記載 | コンポーネント化・変数・命名・Auto Layout |

Figma MCPは「URLを渡せばコードが出てくる魔法」ではありません。レート制限を理解し、大規模デザインは2段階で取得し、Code Connectで既存コンポーネントを再利用する。この3点を押さえるだけで、出力品質と効率が大きく変わります。
