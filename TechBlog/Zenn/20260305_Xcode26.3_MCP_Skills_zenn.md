# Xcode 26.3 の MCP — AI がIDEを「使える」時代

## はじめに

Xcode 26.3 がリリースされました。目玉は **MCP（Model Context Protocol）のネイティブサポート**です。

「MCP」という言葉を聞いても、ピンと来ない方も多いかもしれません。でも、これは iOS 開発のワークフローを根本から変える技術です。

この記事では、以下を解説します:

1. **MCP とは何か** — 技術的に何が起きているのか
2. **なぜ必要なのか** — MCP がない世界で何が辛いのか
3. **どう使うのか** — 7つの Skills で実践する方法

:::message
**公開リポジトリ**
本記事で紹介する Skills はすべて MIT ライセンスで公開しています。
GitHub: [dsgarage/XcodeMCP-Skills](https://github.com/dsgarage/XcodeMCP-Skills)
:::

## MCP がない世界 — 「仲介者」としての開発者

まず、MCP がない世界を考えてみましょう。

AI コーディングアシスタント（ChatGPT、Claude など）に Swift のコードを書いてもらう場面を想像してください。

```
1. AI にコード生成を依頼する
2. 出力されたコードをコピーする
3. Xcode に貼り付ける
4. ビルドする → エラーが出る
5. エラーメッセージをコピーする
6. AI に貼り付けて修正を依頼する
7. 修正コードをコピーする
8. Xcode に貼り付ける
9. またビルドする → まだエラー...
```

**開発者が「仲介者」として、AI と Xcode の間を行き来し続ける**。これが MCP がない世界です。

この手動ループには 3 つの根本的な問題があります:

| 問題 | 詳細 |
|------|------|
| **AI はビルド結果を知らない** | コンパイルエラー、警告、リンクエラー。AI は「コードを渡した後」に何が起きたか知りません |
| **AI はプレビューが見えない** | SwiftUI で UI を書いても、実際にどう表示されるか AI には見えません。開発者がスクリーンショットを撮って送るしかない |
| **コンテキストスイッチの繰り返し** | コピー＆ペーストのたびに集中力が途切れる。これが1日に数十回繰り返される |

**AI は「目も手も持たない優秀な助手」のまま**なのです。

## MCP とは何か — AI に「目と手」を与えるプロトコル

MCP（Model Context Protocol）は、Anthropic が策定したオープンスタンダードです。

一言でいうと: **AI エージェントが外部ツールを直接操作するための通信規約**。

Xcode 26.3 では、この MCP を通じて AI が Xcode を「使える」ようになりました。

### アーキテクチャ

```
開発者
  │ ゴールを伝える
  ▼
Claude Code / Claude Agent
  │ MCP Protocol（JSON-RPC / stdio）
  ▼
mcpbridge（Xcode 同梱バイナリ）
  │ XPC（プロセス間通信）
  ▼
Xcode 26.3 ── 20個のMCPツール
```

重要なポイント:

- **mcpbridge** は Xcode 26.3 に同梱されるバイナリです。別途インストール不要
- **APIキーもクラウドリレーも不要**。ローカルで完結します
- アクティブな Xcode プロセスを**自動検出**して接続します

### MCP がある世界

```
1. 開発者がゴールを伝える:「ログイン画面を作って」
2. AI がコードを書く（XcodeWrite）
3. AI が自分でビルドする（BuildProject）
4. AI がエラーを読む（GetBuildLog）
5. AI が自分で修正する（XcodeUpdate）
6. AI がプレビューを見る（RenderPreview）
7. AI がテストを走らせる（RunSomeTests）
8. AI が「完成しました」と報告する
```

**開発者は「仲介者」から「監督者」になる**。AI が自律的にビルド→修正→検証のループを回し、開発者は結果を確認するだけです。

## Xcode 26.3 が公開する 20 の MCP ツール

Xcode 26.3 は以下の 20 ツールを MCP 経由で公開しています。

### ファイル操作（9ツール）

| ツール | 説明 |
|--------|------|
| `XcodeRead` | ファイル内容を行番号付きで読み取り |
| `XcodeWrite` | ファイルの新規作成・上書き |
| `XcodeUpdate` | テキスト置換による編集（str_replace 方式） |
| `XcodeRM` | ファイル/ディレクトリの削除 |
| `XcodeMV` | 移動・リネーム・コピー |
| `XcodeMakeDir` | ディレクトリ/グループ作成 |
| `XcodeLS` | ディレクトリ内容一覧 |
| `XcodeGlob` | ワイルドカードによるファイル検索 |
| `XcodeGrep` | 正規表現でファイル内テキスト検索 |

### ビルド・テスト（5ツール）

| ツール | 説明 |
|--------|------|
| `BuildProject` | プロジェクトビルド |
| `GetBuildLog` | ビルドログ取得（severity/pattern フィルタ対応） |
| `RunAllTests` | 全テスト実行 |
| `RunSomeTests` | 特定テスト実行（ターゲット＋テスト ID 指定） |
| `GetTestList` | テストプランのテスト一覧取得 |

### 診断・その他（6ツール）

| ツール | 説明 |
|--------|------|
| `XcodeListNavigatorIssues` | Issue Navigator の警告/エラーを構造化データで取得 |
| `XcodeRefreshCodeIssuesInFile` | 特定ファイルのコンパイラ診断 |
| `ExecuteSnippet` | コードスニペットのビルド＆実行（Swift REPL） |
| `DocumentationSearch` | Apple 公式ドキュメント＋WWDC トランスクリプトを検索 |
| `RenderPreview` | SwiftUI プレビューのスクリーンショットを取得 |
| `XcodeListWindows` | Xcode ウィンドウ一覧（tabIdentifier 取得用） |

:::message alert
**前提条件**
MCP 機能を使うには **macOS 26 Tahoe** + **Apple Silicon** が必須です。Intel Mac では利用できません。
:::

## セットアップ — 3 ステップで始める

### Step 1: Xcode MCP を Claude Code に接続

```bash
claude mcp add --transport stdio xcode -- xcrun mcpbridge
claude mcp list   # xcode が表示されれば OK
```

### Step 2: Skills をインストール

```bash
git clone https://github.com/dsgarage/XcodeMCP-Skills.git ~/XcodeMCP-Skills
cd ~/XcodeMCP-Skills
./install.sh --project   # 現在のプロジェクトに配置
```

### Step 3: 使う

Xcode でプロジェクトを開いた状態で:

```bash
claude
> /ios-build-test --save-log   # ビルド＆テスト、結果をログ保存
```

## 7 つの Skills で実践する

### 基本スキル（5つ）

#### 1. ios-build-test — ビルド→テスト→ログ保存

最も基本的なスキルです。ビルドとテストを実行し、結果を `test-results/` に保存します。

```
> /ios-build-test --save-log
```

**内部で起きること:**

```
XcodeListWindows → tabIdentifier 取得
    ↓
BuildProject → ビルド実行
    ↓ (成功)
GetTestList → テスト一覧確認
    ↓
RunAllTests → 全テスト実行
    ↓
XcodeWrite → test-results/2026-03-05_143022.md にログ保存
```

テスト結果はプロジェクト内にマークダウン形式で保存されます:

```markdown
# テスト結果 - 2026-03-05 14:30:22

## サマリー
- スキーム: MyApp
- 結果: PASS
- 合計: 42 / 成功: 42 / 失敗: 0

## ビルドログ（警告）
LoginView.swift:23 - 'onChange(of:perform:)' is deprecated
```

#### 2. ios-preview-check — SwiftUI プレビューの視覚検証

AI が `RenderPreview` でスクリーンショットを取得し、**実際の見た目を確認**しながらコードを修正します。

```
> /ios-preview-check Sources/Views/LoginView.swift
```

これが MCP の真価です。AI がプレビューを「見て」判断できる。レイアウト崩れ、文字切れ、ダークモード対応漏れを AI が自分で検出し、修正→再プレビューのループを回します。

#### 3. ios-doc-fix — Deprecated API の自動修正

`DocumentationSearch` で Apple 公式ドキュメント（WWDC トランスクリプト含む）を検索し、代替 API を特定して一括修正します。

```
> /ios-doc-fix SwiftUI
```

**内部フロー:**

```
XcodeListNavigatorIssues(severity: "warning", pattern: "deprecated")
    ↓ deprecated 警告を収集
DocumentationSearch("代替API名")
    ↓ 公式ドキュメントで移行先を確認
XcodeGrep → 使用箇所を全数把握
    ↓
XcodeUpdate → 一括修正
    ↓
BuildProject → ビルド検証
```

#### 4. ios-file-ops — ファイル操作のベストプラクティス

Xcode MCP の 9 つのファイル操作ツールを安全に使うためのガイドスキルです。特に重要なのは:

- `XcodeWrite` は**ターゲットに自動追加されない**（Xcode 上で手動追加が必要な場合あり）
- `XcodeRM` は**破壊的操作**。必ずユーザー確認を取る
- ファイル移動後は**必ずビルド検証**する

#### 5. ios-diagnostics — ビルドエラーの体系的診断

ビルドエラーを分類し、依存関係順に修正します。

| カテゴリ | 対処法 |
|----------|--------|
| 参照エラー | import 追加 or 宣言確認 |
| 型エラー | 型の修正 |
| 構文エラー | コード修正 |
| 依存エラー | パッケージ設定確認 |

`ExecuteSnippet` で修正した API の動作を即座に検証できるのも便利です。

### /simplify & /batch 拡張スキル（2つ）

Claude Code v2.1.63 で追加された `/simplify` と `/batch` は強力なスキルですが、標準では Xcode MCP ツールを活用しません。そこで、iOS 開発に特化した拡張版を作りました。

:::message
`/simplify` と `/batch` の基本的な使い方は、以下の記事で詳しく解説しています:
[Claude Code新スキル /simplify と /batch](https://zenn.dev/dsgarage/articles/claude-code-simplify-batch)
:::

#### 6. ios-simplify — PR 前の最終仕上げ

標準の `/simplify` は 3 つの並列エージェントでコードレビューします:

| エージェント | チェック内容 |
|-------------|------------|
| **Reuse** | 重複コードの検出・統合 |
| **Quality** | 命名・構造・制御フローの品質 |
| **Efficiency** | 不要な処理、見落とした最適化 |

`/ios-simplify` はこの 3 観点に加え、**Xcode MCP で iOS 固有の検証**を追加します:

```
/simplify 標準レビュー（3エージェント並列）
    ↓ 修正適用
BuildProject         → コンパイル通るか？
    ↓
XcodeListNavigator   → 警告が増えていないか？
Issues
    ↓
RenderPreview        → SwiftUI の見た目が壊れていないか？
    ↓
RunSomeTests         → テストがパスするか？
    ↓
DocumentationSearch  → deprecated API を新たに導入していないか？
    ↓
レポート出力 → git diff 確認 → commit → PR
```

**推奨ワークフロー:**

```
実装完了 → git commit → /ios-simplify → git diff 確認 → git commit → PR
```

標準の `/simplify` との違いは明確です。「コードレビューで OK」だけでなく、**「ビルドが通る」「テストがパスする」「UI が崩れていない」まで保証**します。

#### 7. ios-batch — 大規模マイグレーションの並列実行

50 ファイルに散らばる deprecated API の更新。UIKit から SwiftUI への段階移行。手作業なら丸一日かかるマイグレーションを、`/ios-batch` は並列で実行します。

```
> /ios-batch UIKit の UIAlertController を全て SwiftUI の .alert に移行
```

**4 フェーズで実行:**

```
Phase 1: 調査
  XcodeGrep + XcodeGlob で影響範囲を特定
  DocumentationSearch で移行先 API を確認

Phase 2: 計画
  5〜30 の独立ユニットに分解
  ユーザーに計画を提示して承認を得る

Phase 3: 並列実行
  各ワーカーが独立した git worktree で:
    XcodeUpdate → BuildProject → RunSomeTests → /ios-simplify → PR 作成

Phase 4: 完了レポート
  成功/失敗のステータステーブルを表示
```

**iOS 固有のガードレール** が最大の特徴です:

| 変更禁止ファイル | 理由 |
|-----------------|------|
| `*.pbxproj` | Xcode プロジェクト設定の破損リスク |
| `Info.plist` | アプリ設定への意図しない影響 |
| `*.entitlements` | 権限設定は別作業 |
| `*.xcdatamodeld` | Core Data はマイグレーション戦略が必要 |
| `*.storyboard` / `*.xib` | IB ファイルの自動変更は参照破壊のリスク |

これらのファイルに変更が必要な場合、ワーカーはスキップしてユーザーに手動対応を報告します。

## まとめ — MCP がもたらす変化

| 項目 | MCP なし | MCP あり |
|------|---------|---------|
| ビルド検証 | 手動コピペ | AI が `BuildProject` で自動実行 |
| エラー修正 | エラーメッセージを手動で AI に伝達 | AI が `GetBuildLog` で自分で読んで修正 |
| UI 確認 | スクリーンショットを撮って送る | AI が `RenderPreview` で自分で見る |
| テスト | 手動実行して結果を伝える | AI が `RunSomeTests` で自分で実行 |
| ドキュメント | 自分で調べて AI に伝える | AI が `DocumentationSearch` で自分で調べる |

**MCP は AI に「目と手」を与える技術**です。

開発者の役割は「仲介者」から「監督者」へ。AI がビルド→修正→検証のループを自律的に回し、開発者は方向性の指示と最終確認に集中できる。
