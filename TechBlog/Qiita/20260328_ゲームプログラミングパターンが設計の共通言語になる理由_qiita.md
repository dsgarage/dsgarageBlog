# ゲームプログラミングパターンはなぜ「設計の共通言語」になるのか

「Commandパターンで入力を抽象化しよう」——この一言で、チーム全員が同じ設計像を頭に描けるだろうか。パターンの名前は知っている。GoF本も読んだ。しかし、レビューで「ここはObserverでいけるよね」と言われたとき、ゲーム開発の文脈でそれが何を意味するのか、即座に共有できるチームはそう多くない。

Robert Nystromの『Game Programming Patterns』は、GoFのデザインパターンをゲーム開発の現場に再解釈し、さらにゲーム固有の13パターンを体系化した一冊だ。本書の価値は「パターンのカタログ」にとどまらない。パターンという共通語彙を通じて、プログラマ同士が設計意図を正確に伝え合える——つまり「設計の共通言語」を手に入れることにある。

本記事では、この書籍の核心パターンを軸に、なぜパターンが「知識」から「言語」へと変わるのかを解説する。読み終えた後、あなたはパターン名を使って設計会話ができるようになる。

---

## 背景: なぜゲーム開発にパターンが必要なのか

ゲームコードには、ビジネスアプリケーションにはない固有の難しさがある。Nystromは本書の冒頭で4つの課題を挙げている。

1. **時間とシーケンシング**: ゲームでは「正しい順序で、正しいタイミングで」処理が走る必要がある
2. **並行開発の圧力**: 短い開発サイクルの中で、複数のプログラマが互いの領域を踏まずに機能を追加しなければならない
3. **相互作用の爆発**: モンスターがヒーローを噛み、ポーションが混合され、爆弾が敵味方を吹き飛ばす——これらの相互作用がコードを絡み合わせる
4. **パフォーマンス**: フレーム落ちは即座にユーザー体験を損なう。1サイクルの節約が勝負を分ける

これらの課題に対して、場当たり的に対処するとコードは「巨大な毛玉（giant hairball）」になる。Nystromがかつて Electronic Arts で目にしたのは、グラフィクスやAIの個々のコードは優秀なのに、それを支えるアーキテクチャが後回しにされている現実だった。

パターンは、この構造的な問題に名前をつける。名前があれば、議論できる。議論できれば、改善できる。

---

## 理論: 本書の20パターンを俯瞰する

『Game Programming Patterns』は20のパターンを5つのカテゴリに分類している。そのうち6つはGoFパターンのゲーム向け再解釈、残り14はゲーム開発固有のパターンだ。

![GoFパターン vs ゲーム特有パターンの分類図](https://raw.githubusercontent.com/dsgarage/dsgarageBlog/main/TechBlog/images/gpp_pattern_classification.png)

| カテゴリ | パターン数 | 代表例 | 解決する問題 |
|:---|:---|:---|:---|
| **GoF再解釈** | 6 | Command, Observer, State | 入力・通知・状態管理 |
| **シーケンシング** | 3 | Game Loop, Update Method | 時間制御・フレーム更新 |
| **ビヘイビア** | 3 | Bytecode, Type Object | データ駆動の振る舞い |
| **デカップリング** | 3 | Component, Event Queue | ドメイン分離 |
| **最適化** | 5 | Data Locality, Object Pool | メモリ・キャッシュ効率 |

重要なのは、これらが独立したレシピではなく、互いに連携する点だ。Game Loopの中でUpdate Methodが呼ばれ、Update MethodのなかでStateパターンが状態遷移を管理し、Observerが変化を通知する。パターンを「組み合わせの語彙」として持つことで、設計全体を言語化できるようになる。

### 核心概念1: Commandパターン——「操作のオブジェクト化」

Nystromの端的な定義はこうだ。

> A command is a reified method call.（コマンドとは、メソッド呼び出しを「モノ」にしたものだ）

GoFの原典では「リクエストをオブジェクトとしてカプセル化する」と定義されるが、ゲーム開発ではより具体的な意味を持つ。

![Commandパターンのフロー図](https://raw.githubusercontent.com/dsgarage/dsgarageBlog/main/TechBlog/images/gpp_command_pattern_flow.png)

**入力マッピングの抽象化**: ボタン押下を直接 `jump()` に紐づけるのではなく、`JumpCommand` オブジェクトを生成する。これにより、キーコンフィグの変更がコード変更なしで実現できる。

**Undo/Redo**: コマンドを履歴スタックに積めば、レベルエディタのUndo機能やリプレイシステムが自然に実装できる。

**AIの再利用**: 同じ `Command` インターフェースをAIからも発行できるため、プレイヤーとAIで入力処理コードを共有できる。

```csharp
// Unity での Command パターン適用例
public interface ICommand {
    void Execute(GameActor actor);
    void Undo(GameActor actor);
}

public class JumpCommand : ICommand {
    private Vector3 _previousPosition;

    public void Execute(GameActor actor) {
        _previousPosition = actor.transform.position;
        actor.Jump();
    }

    public void Undo(GameActor actor) {
        actor.transform.position = _previousPosition;
    }
}
```

チームで「入力システムはCommandで切ろう」と言えば、全員が上記の構造を想起できる。これがパターンが「言語」として機能する瞬間だ。

### 核心概念2: Stateパターン——「フラグ地獄からの解放」

Nystromが本書で最も生々しく描写するのが、State パターンの動機となる「フラグ地獄」だ。

プラットフォーマーのヒロインに「立ち」「ジャンプ」「しゃがみ」「ダイブ」の動作を追加していくとする。`isJumping_`、`isDucking_` といったBooleanフラグを追加するたびにバグが増殖する。ジャンプ中にしゃがみボタンを押すとダイブになるが、ダイブ中に空中ジャンプを防ぐフラグが漏れる——この連鎖が止まらない。

解決策は有限状態マシン（FSM）だ。状態を明示的なオブジェクトとして切り出し、遷移ルールを宣言的に定義する。

```csharp
// Unreal Engine の GameplayAbilitySystem も内部的にはステートマシンの考え方
// Unity での簡易 State パターン
public abstract class HeroineState {
    public abstract HeroineState HandleInput(Heroine heroine, Input input);
    public abstract void Update(Heroine heroine);
}

public class StandingState : HeroineState {
    public override HeroineState HandleInput(Heroine heroine, Input input) {
        if (input == Input.PressB) return new JumpingState();
        if (input == Input.PressDown) return new DuckingState();
        return this;
    }
    // ...
}
```

「この挙動はStateで管理している」と言えば、レビュアーは状態遷移図を想像し、遷移の抜け漏れを指摘できる。フラグの組み合わせを追いかける必要はない。

### 核心概念3: Component——「モノリスを分割する」

ゲームキャラクターは入力・物理・描画・サウンドなど複数のドメインにまたがる。これをひとつのクラスに押し込むと5,000行の「ダンピンググラウンド」が生まれる——Nystromの言葉を借りれば、「勇敢な忍者コーダーだけが足を踏み入れる」コードになる。

Componentパターンは、このモノリスをドメイン境界で切り分ける。

![Component / ECS の構造図](https://raw.githubusercontent.com/dsgarage/dsgarageBlog/main/TechBlog/images/gpp_component_ecs.png)

このパターンは「継承よりコンポジション」の原則を体現している。Nystromはレストランのメニューに例える。モノリシッククラスは「セットメニューしか注文できない状態」だが、Componentは「アラカルト」——必要な機能だけを組み合わせられる。

**Unityとの対応**: UnityのGameObject + MonoBehaviour アーキテクチャは、まさにこのComponentパターンの実装だ。`GetComponent<T>()` でコンポーネントを取得し、`AddComponent<T>()` で動的に追加する設計は、本書の概念がそのままエンジンに組み込まれている例である。

**ECSへの発展**: さらに進んだ Entity-Component-System（ECS）では、Entityは単なるID、Componentはデータのみ、Systemがロジックを担当する。Unity DOTSやUnreal Engine の Mass Entity はこのアーキテクチャを採用している。ComponentパターンからECSへの流れを知っていれば、エンジンの設計思想を深く理解できる。

### 核心概念4: Game Loop + Update Method——「ゲームの心臓」

ゲームと他のソフトウェアを分かつ最も根本的な違い——それは「ユーザー入力がなくても世界が動き続ける」ことだ。

![Game Loop + Update Method の関係図](https://raw.githubusercontent.com/dsgarage/dsgarageBlog/main/TechBlog/images/gpp_gameloop_update.png)

Game Loopは `processInput() → update() → render()` を無限に回し続ける。Update Methodは、そのupdate()の中で個々のオブジェクトに1フレーム分の振る舞いを委譲する。

Nystromが巧みに描くのは、この2つのパターンがないとどうなるかだ。スケルトン兵のパトロールを `for` ループで書けば、ゲームはその間フリーズする。Game Loopに毎フレームの更新を委ねることで初めて、複数のオブジェクトが「同時に」動いているように見える。

| エンジン | Game Loop | Update Method |
|:---|:---|:---|
| **Unity** | `PlayerLoop` が内部管理 | `MonoBehaviour.Update()` |
| **Unreal** | `FEngineLoop::Tick()` | `AActor::Tick()` |
| **Godot** | `MainLoop` | `Node._process()` |

「Update()の中で重い処理をするな」「FixedUpdate()で物理を回せ」——こうしたチーム内の会話は、Game LoopとUpdate Methodの関係を全員が理解していて初めて意味を持つ。

## 分析: パターンが「共通言語」になる3つの条件

パターンを知っているだけでは「言語」にはならない。チーム内で共通言語として機能するには、3つの条件が必要だ。

### 条件1: 名前が問題と解決策を同時に指す

「Observerパターン」と言えば、「あるオブジェクトの変化を、依存関係なしに他のオブジェクトへ通知する仕組み」が一語で伝わる。問題（密結合な通知）と解決策（サブジェクト/オブザーバーの分離）が名前に圧縮されている。

### 条件2: 実装の詳細ではなく意図を伝える

「ここはCommandで」と言うとき、具体的なクラス構造を指定しているのではない。「操作をオブジェクト化してUndoや再利用を可能にする意図」を伝えている。C++でもC#でもRustでも、同じ意図を共有できる。

### 条件3: トレードオフが共有される

Nystromは各パターンに「Keep in Mind」セクションを設けている。Componentパターンは柔軟だが、間接参照のコストがかかる。Singletonは便利だが、グローバル状態の罠がある。パターン名を出すだけで、そのトレードオフまで含めた議論が始まる。

## 実践: あなたのプロジェクトに適用する

### ステップ1: 現在のコードの「匂い」を嗅ぐ

以下の症状があれば、対応するパターンの適用を検討する。

| コードの匂い | 候補パターン |
|:---|:---|
| 巨大なswitch文で入力を処理 | Command |
| 変更のたびに複数クラスを修正 | Observer |
| Booleanフラグが3つ以上の組み合わせ | State |
| 1クラスが複数ドメインにまたがる | Component |
| ゲームオブジェクトの生成/破棄が頻繁 | Object Pool |

### ステップ2: パターンをチームの語彙にする

コードレビューで「ここはCommandで抽象化できそう」と言える文化を作る。パターン名をプルリクエストの説明に使う。設計ドキュメントにパターンの分類図を貼る。

### ステップ3: 段階的に導入する

Nystromは本書を「アラカルト」と表現している。20パターンすべてを一度に導入する必要はない。まずCommand・Observer・Stateの3つから始めれば、多くの設計会話がカバーできる。

## よくある落とし穴

**パターンの過剰適用**: 「使えるから使う」は危険だ。Nystromも「パターンを使わないことが最良の選択になることもある」と繰り返し警告している。5行で済むコードにCommandパターンを適用すれば、複雑さが増すだけだ。

**パターン名の一人歩き**: 「Singletonで」と言ったとき、チームメンバーがGoFのSingletonを想像するのか、Service Locatorを想像するのかがずれていれば、共通言語として機能しない。本書を読み合わせることで、認識を揃える工程が必要になる。

**パフォーマンスの見落とし**: 本書の後半には Data Locality や Object Pool といった最適化パターンがある。設計の美しさだけを追求して、ゲーム特有のパフォーマンス要件を忘れてはならない。60FPSを維持できないアーキテクチャは、どんなに美しくても失敗だ。

---

## まとめ

デザインパターンの本質は、コードの再利用ではなく「設計意図の再利用」にある。『Game Programming Patterns』が提供するのは、ゲーム開発者が共有できる20個の語彙だ。Commandと言えば入力の抽象化、Stateと言えばFSMによる状態管理、Componentと言えばドメイン分割——この圧縮された言語があるからこそ、チームは設計について効率よく議論できる。
