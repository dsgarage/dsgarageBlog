# 執筆品質ルール

## リード文の3要素（必須）

すべての記事のリード文に以下を含める:
1. **問題提起** — 読者が直面する具体的な状況
2. **具体例** — 実際のプロジェクトや数値を含む例
3. **この記事の価値** — 読むことで得られるメリット

## 本文の構成順序

```
1. 問題の提示（できること vs できないこと）
2. なぜそうなるのか（技術的背景）
3. 具体的な影響（作業フロー比較）
4. 解決策の提示（または次回への導線）
5. まとめ
```

## 品質基準

- **視覚的**: 図・コード・表で結果が見える
- **教育的**: 「なぜ」が説明されている
- **実用的**: 具体的な手順がある
- **最小限**: 冗長な説明がない
- **段階的**: 基礎から応用へ

## 表記ルール

- 専門用語の初出には必ず説明を付ける（例: MCP（Model Context Protocol））
- コードブロックには何をするコードか明記する
- 「便利です」「効率的です」で終わらず、具体的な数値で示す
- 各記事は単独で完結させる（「前回の記事を読んでください」は禁止）

## 図解の優先順位

1. **Graphviz**（第一選択）— `images/_style.dot` のカラーパレットに準拠
2. **比較表**（対比が必要な場合）
3. **ASCII図**（コードブロック内での簡易説明のみ）
