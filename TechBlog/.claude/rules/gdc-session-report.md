# GDC セッションレポート執筆ルール

GDC セッションレポートは通常のブログ記事とは異なるフォーマットに従う。

## 記事の構成順序（必須）

```
1. リード文（ゲーム/技術の課題 → セッション概要 → 本記事の価値）
2. 技術的位置づけ（使用エンジン、Unity との関係）
3. セッション概要（表形式）
4. 技術セクション × N（問題 → スライド詳細 → 解決策 → 結果）
5. まとめ（教訓 3〜5 ポイント）
6. Unity エンジニア対応ガイド（※非 Unity セッションの場合）
7. 参考情報（表形式）
```

## スライド書き起こしルール

1. スライドのタイトルを **太字** で明記
2. 箇条書き・ラベル・注釈を **全文転記**（省略禁止）
3. コードが写っている場合は **言語指定付きコードブロック** で転記
4. 図の構成要素（左右の比較、色分け、矢印の意味）を文章で説明
5. スピーカーの口頭補足（トランスクリプトから）があれば引用で追記

## Unity 対応ガイドの 3 段構成（各セクション必須）

```
1. **元エンジンで発生していた問題**: 具体的な症状・数値・原因
2. **Unity で同様の問題が起きるケース**: 3〜5 個の具体的シナリオ
3. **Unity でのアプローチ + 具体的なアクション**: API・クラス・コード例
```

## Unity API 検証チェックリスト

記事完成後、以下を必ず確認する:

- [ ] クラス名・メソッド名が公式リファレンスに存在するか
- [ ] 名前空間が Unity 6 の最新に合っているか（Experimental → 正式移行に注意）
- [ ] 仕様値（ライト上限、バッファサイズ等）が正確か
- [ ] `#pragma` ディレクティブの記法が正しいか
- [ ] コード例のシグネチャが実際の API と一致するか

## 図解（draw.io）ルール

- **最低 2〜3 枚** のオリジナル図解を作成する
- データ構造の比較、パイプラインフロー、時系列動作が優先
- `.drawio` 元ファイルを `slides/` に保存（後から編集可能に）
- `drawio --export --format png --scale 2` で PNG 化
- 図のカラーパレット: 赤=問題、緑=解決策、青=処理、黄=注目、紫=特殊処理

## 画像パスの注意

PDF 生成（weasyprint）は TechBlog/ ディレクトリで実行される。
画像パスは **TechBlog/ からの相対パス** にすること:

```markdown
<!-- ✅ 正しい -->
![説明](../stock/GDC2026/SessionName/slides/IMG_XXXX.jpg)

<!-- ❌ 間違い（PDF で画像が表示されない） -->
![説明](slides/IMG_XXXX.jpg)
```

## 禁止事項

- UniMCP4CC 誘導は GDC 記事には含めない
- トランスクリプトにない情報を創作しない
- スピーカーの発言を改変しない（意訳は可、捏造は不可）
- 「便利です」「効率的です」で終わらず、具体的な数値で示す
