# UnityでAIを使う時の壁

「AIでゲーム開発が効率化される」——そう信じて Claude Code を導入しました。
C#スクリプトは確かに数秒で生成されます。しかし、その直後に気づきます。

**「このコードを動かすまでに、まだ5分かかる」**

スクリプトを Project に保存し、GameObject を作成し、コンポーネントを追加し、参照を設定し……。
この手作業は、AIがどれだけ優秀になっても変わりません。

本記事では、Unity開発においてAIだけでは解決できない「壁」の正体を解説します。
この壁を理解することで、適切なツール選択と作業フローの設計ができるようになります。

:::message
**この問題を解決するツールがあります**

本記事で説明する「AIでUnity Editorを操作できない」という問題は、**UniMCP4CC**（Unity MCP Server for Claude Code）で解決できます。

- GitHub: [dsgarage/UniMCP4CC](https://github.com/dsgarage/UniMCP4CC)
- 対応Unity: 2021.3 LTS以降
- ライセンス: MIT
:::

## AIが得意なこと：スクリプト生成

AIは、C#スクリプトを迅速に生成できます。
設計意図を伝えるだけで、数秒で「とりあえず動く形」のコードが手に入ります。

```csharp
// AIが生成したPlayerController（例）
public class PlayerController : MonoBehaviour
{
    [SerializeField] private float moveSpeed = 5f;
    [SerializeField] private float jumpForce = 10f;

    private Rigidbody rb;

    void Start()
    {
        rb = GetComponent<Rigidbody>();
    }

    void Update()
    {
        float horizontal = Input.GetAxis("Horizontal");
        float vertical = Input.GetAxis("Vertical");
        Vector3 movement = new Vector3(horizontal, 0, vertical) * moveSpeed;
        rb.velocity = new Vector3(movement.x, rb.velocity.y, movement.z);
    }
}
```

このコードは数秒で生成されます。
問題は、**ここから先**にあります。

## AIができないこと：Unity Editorの操作

### Unity EditorはGUIアプリケーション

Unity Editorは**グラフィカルユーザーインターフェース（GUI）**で動作するアプリケーションです。
マウスとキーボードによる操作が前提となっています。

```
┌──────────────────────────────────────────────────────┐
│                    Unity Editor                       │
│  ┌─────────────┐ ┌─────────────┐ ┌───────────────┐  │
│  │  Hierarchy  │ │    Scene    │ │   Inspector   │  │
│  │             │ │             │ │               │  │
│  │ ・Main Cam  │ │   [3D View] │ │ Transform     │  │
│  │ ・Light     │ │             │ │ Position: ... │  │
│  │ ・Player    │ │             │ │ Rotation: ... │  │
│  │             │ │             │ │               │  │
│  └─────────────┘ └─────────────┘ └───────────────┘  │
│                                                       │
│  ※ マウス・キーボードでの操作が前提                  │
└──────────────────────────────────────────────────────┘
```

AIはこの画面を「見る」ことも「操作する」こともできません。
これが、Unity開発におけるAIの根本的な限界です。

### 具体的にできないこと

以下の作業は、すべて人間が手動で行う必要があります：

| 作業 | 内容 | なぜできないか |
|:---|:---|:---|
| **Scene構築** | Hierarchy内のオブジェクト配置 | ファイル操作だけでは不可能 |
| **GameObject作成** | 作成・削除・親子関係設定 | Editor API呼び出しが必要 |
| **Component追加** | Rigidbody、Collider等 | Inspectorでの操作が必要 |
| **参照設定** | SerializeFieldへの参照設定 | ドラッグ&ドロップが必要 |
| **数値調整** | Transform、物理パラメータ | 実行時の値変更が必要 |
| **Prefab化** | アセット作成・管理 | AssetDatabase操作が必要 |

## 問題の本質：コードと設定の分離

### Unityの設計思想

Unityは、**コード（ロジック）** と **設定（データ）** を分離する設計思想を持っています。
これは柔軟で安全な開発を可能にする、とても健全なアーキテクチャです。

```
┌────────────────────────────────────────────────┐
│              Unityのアーキテクチャ              │
│                                                │
│  ┌──────────────┐       ┌──────────────┐      │
│  │    コード    │       │     設定     │      │
│  │  (.csファイル)│       │  (Inspector) │      │
│  │              │       │              │      │
│  │ ・ロジック   │       │ ・Transform  │      │
│  │ ・アルゴリズム│       │ ・Component  │      │
│  │ ・振る舞い   │       │ ・参照       │      │
│  └──────┬───────┘       └───────┬──────┘      │
│         │                       │              │
│         └──────────┬────────────┘              │
│                    ▼                           │
│           ┌──────────────┐                    │
│           │  動くゲーム  │                    │
│           └──────────────┘                    │
│                                                │
│  ※ 両方が揃って初めて「動く」                 │
└────────────────────────────────────────────────┘
```

### AIが書けるのはコードだけ

この設計において、AIがテキストファイルとして生成できるのは「コード」だけです。
「設定」はUnity Editorを通じて行う必要があります。

```
AIができること:
└── .csファイルを生成する

AIができないこと:
├── Sceneにオブジェクトを配置する
├── コンポーネントを追加する
├── 参照を設定する
└── パラメータを調整する
```

つまり、**AIがどれだけ優れたコードを書いても、そのコードを「動く状態」にするには人間の手作業が必要**なのです。

## 従来のワークフローの問題点

### 典型的な作業フロー

AIでスクリプトを生成した後、以下の手動作業が必要になります：

```
┌─────────────────────────────────────────────────────┐
│                  従来のワークフロー                  │
│                                                     │
│  1. AIに指示  ─────────────→  スクリプト生成       │
│     「Playerを作って」         PlayerController.cs  │
│                                                     │
│  2. 手動作業  ─────────────→  何度も繰り返し       │
│     ├── Projectに配置                              │
│     ├── GameObjectを作成                           │
│     ├── スクリプトをアタッチ                       │
│     ├── Rigidbodyを追加                            │
│     ├── Colliderを追加                             │
│     ├── パラメータを設定                           │
│     └── 参照をドラッグ&ドロップ                    │
│                                                     │
│  ※ ここに時間がかかる                             │
└─────────────────────────────────────────────────────┘
```

### 時間の非対称性

具体的な時間を比較すると、問題の深刻さが分かります：

```
コード生成:     5秒
手動セットアップ: 5分

→ コード生成は全体の2%にすぎない
→ 実質的な効率化: ほぼゼロ
```

### 創造性への影響

この「試すまでの距離」が長いほど、以下の問題が発生します：

1. **試行回数が減る** - 「面倒だからあとで」と先延ばしにする
2. **アイデアを削る** - 「これは複雑すぎるからやめよう」と諦める
3. **可能性を諦める** - 「時間がないから簡単な方法で」と妥協する

つまり、**手動作業のコストが、創造性そのものを制限している**のです。

## バッチモードの限界

「Unity Editor操作が必要なら、スクリプトで自動化すればいいのでは？」

この発想は自然ですが、現実には多くの問題があります。

### エディタスクリプトの例

```csharp
// エディタスクリプトで自動化する場合
using UnityEditor;
using UnityEngine;

public class SceneSetup
{
    [MenuItem("Tools/Setup Scene")]
    public static void Setup()
    {
        var player = new GameObject("Player");
        player.AddComponent<Rigidbody>();
        player.AddComponent<CapsuleCollider>();

        EditorSceneManager.SaveScene(
            EditorSceneManager.GetActiveScene()
        );
    }
}
```

### なぜこれでは不十分なのか

| 問題 | 説明 |
|:---|:---|
| **事前準備が必要** | 「今思いついた操作」には対応できない |
| **GUI操作が必要** | 結局メニューから実行する必要がある |
| **対話的ではない** | 「ちょっと位置を変えて」ができない |
| **コード量が増える** | 簡単な操作も大量のコードが必要 |

### 具体的な時間比較

```
やりたいこと: 「Playerを作って、位置を(0, 1, 0)にして」

バッチモードの場合:
├── スクリプトを書く（10分）
├── コンパイルを待つ（30秒）
├── メニューから実行（5秒）
├── 結果を確認（5秒）
├── 位置が違ったので修正（10分）
└── 再度コンパイル＆実行...

理想の状態:
└── 「はい、作りました」（5秒）
```

バッチモードでは、**対話的な調整ができない**ことが最大の問題です。

## 解決策：MCP（Model Context Protocol）

この問題を解決するのが、**MCP（Model Context Protocol）**です。

### MCPとは

MCPは、AIとUnity Editorの間を**橋渡しする仕組み**です。
AIからの自然言語の指示を、Unity Editor APIの呼び出しに変換します。

```
┌─────────────────────────────────────────────────────┐
│                 解決のアーキテクチャ                │
│                                                     │
│  ┌─────────┐                      ┌─────────────┐  │
│  │   AI    │                      │Unity Editor │  │
│  │         │                      │             │  │
│  │「Player │                      │  Hierarchy  │  │
│  │  を作成  │                      │  Inspector  │  │
│  │  して」 │                      │  Scene      │  │
│  └────┬────┘                      └──────┬──────┘  │
│       │                                  │          │
│       │   ┌────────────────────┐        │          │
│       └──→│       MCP          │←───────┘          │
│           │                    │                   │
│           │  ・指示を受け取る  │                   │
│           │  ・APIに変換する   │                   │
│           │  ・Editorを操作    │                   │
│           │  ・結果を返す      │                   │
│           └────────────────────┘                   │
│                                                     │
└─────────────────────────────────────────────────────┘
```

### MCPがもたらす変化

MCPを使うと、自然言語でUnity Editorを操作できます：

```
あなた: 「Playerを作成して、Rigidbodyを追加して、位置を(0, 1, 0)にして」
Claude: 「完了しました」
```

**350以上のAPI**で、ほぼすべてのEditor操作をカバーしています。

### 時間比較

```
従来のワークフロー:
├── コード生成: 5秒
├── 手動セットアップ: 5分
└── 合計: 約5分

MCPを使ったワークフロー:
├── コード生成: 5秒
├── Editor操作（MCP経由）: 30秒
└── 合計: 約35秒
```

**5分が35秒に短縮**——これがMCPの価値です。

## まとめ

### Unity開発における「壁」

| 問題 | 説明 |
|:---|:---|
| **GUIアプリ** | Unity Editorはマウス操作が前提 |
| **AIの限界** | AIはGUIを「見る」ことも「操作する」こともできない |
| **コードと設定の分離** | コードだけでは「動く状態」にならない |
| **手動作業** | 毎回のセットアップ作業が必要 |

### 結果として起きること

- 試行回数が減る
- アイデアを諦める
- **創造性が制限される**

### 解決策

**MCP（Model Context Protocol）** を使えば、AIがUnity Editorを直接操作できます。
手作業5分が30秒に短縮され、思いついたアイデアをすぐに試せるようになります。
